# E01: Baseline Conversion Ranking

## 1. Real-World Problem

A sales team receives many leads, but it cannot contact everyone. Each day or time window, the team has a fixed capacity:

\[
B
\]

For example, if there are 100,000 leads and the sales team can only contact 500, the system must choose which 500 leads should be sent to sales.

The simplest business question is:

> Which leads are most likely to convert?

This first experiment builds that baseline. It does not try to estimate causal uplift yet. It only tests whether a normal conversion model can rank leads better than random selection.

This experiment is important because it creates the benchmark that later uplift models must beat.

---

## 2. Mathematical Definition

For each lead \(i\), define:

| Symbol | Meaning |
|---|---|
| \(X_i\) | features of lead \(i\) |
| \(Y_i \in \{0,1\}\) | observed conversion label |
| \(\hat{s}_i\) | predicted conversion score |
| \(B\) | sales capacity / top-K budget |
| \(S_B\) | selected set of \(B\) leads |

The baseline model estimates:

\[
\hat{s}_i = \hat{P}(Y_i=1 \mid X_i)
\]

The baseline allocation policy selects the top \(B\) leads by predicted conversion score:

\[
S_B^{score}
=
\operatorname{TopB}_i(\hat{s}_i)
\]

Equivalently, the policy solves:

\[
\max_{A_i}
\sum_i A_i \hat{s}_i
\]

subject to:

\[
\sum_i A_i \le B
\]

\[
A_i \in \{0,1\}
\]

where:

\[
A_i =
\begin{cases}
1 & \text{if lead } i \text{ is selected} \\
0 & \text{otherwise}
\end{cases}
\]

If every selected lead consumes the same sales capacity, the optimal solution is simply:

> Sort leads by \(\hat{s}_i\), then take the top \(B\).

---

## 3. Premise, Assumptions, and Scope

### Premise

The premise of this experiment is:

> A supervised conversion model should concentrate more actual converters in the top \(B\) leads than random selection.

This is a prediction and ranking experiment, not a causal experiment.

It estimates:

\[
P(Y=1 \mid X)
\]

not:

\[
Y(1)-Y(0)
\]

So it answers:

> Who is likely to convert?

It does not answer:

> Who converts because sales contacted them?

### Assumptions

**A1. Labels are observed after the feature window.**

Features \(X_i\) must be measured before the conversion outcome \(Y_i\).

This avoids temporal leakage.

**A2. The prediction target is well-defined.**

For binary conversion:

\[
Y_i =
\begin{cases}
1 & \text{if lead } i \text{ converts within the outcome window} \\
0 & \text{otherwise}
\end{cases}
\]

The outcome window should be fixed, for example 7 days, 14 days, or 30 days after lead creation.

**A3. Sales capacity is fixed.**

The model is evaluated at top \(B\), not at an arbitrary probability threshold.

This matches the operational constraint:

\[
\sum_i A_i \le B
\]

**A4. All leads have equal contact cost in this first experiment.**

This experiment ignores differences in rep time, discount cost, lead value, and action type. Those are introduced later in profit-aware and uplift-aware allocation.

**A5. No causal claim is made.**

This experiment may show that the model finds likely converters, but it cannot prove that contacting those leads causes conversion.

---

## 4. Why This Experiment Matters

This baseline is necessary for four reasons.

### 4.1 It matches the current industry default

Most lead scoring systems estimate:

\[
P(Y=1 \mid X)
\]

Then they send the highest-scoring leads to sales.

This is a reasonable first system because it is simple, measurable, and operationally useful.

### 4.2 It reframes classification as ranking

In normal classification, the model predicts a label:

\[
\hat{Y}_i \in \{0,1\}
\]

But in sales operations, the team usually does not need a hard label. It needs a prioritized queue.

So the important output is a ranking:

\[
\hat{s}_{(1)} \ge \hat{s}_{(2)} \ge \cdots \ge \hat{s}_{(n)}
\]

The top part of the ranking matters more than the bottom part.

### 4.3 It handles class imbalance better than accuracy

Lead conversion is usually rare. If only 1 percent of leads convert, a model that predicts "no conversion" for everyone gets 99 percent accuracy but is useless for sales.

Therefore, this experiment should not use accuracy as the main metric.

It should use top-K ranking metrics.

### 4.4 It creates the comparison point for uplift

Later experiments will test uplift policies:

\[
\hat{\tau}(X_i)=\hat{E}[Y_i(1)-Y_i(0)\mid X_i]
\]

Those policies should be compared against this baseline. The key question later is:

> Does uplift allocation create more incremental value than conversion ranking?

## 5. How To Conduct the Experiment

### Step 1: Prepare the dataset

Use one row per lead.

Minimum columns:

| Column | Meaning |
|---|---|
| `lead_id` | unique lead identifier |
| feature columns | pre-conversion lead features |
| `converted` | binary outcome \(Y_i\) |
| `created_at` | lead creation time |
| `outcome_time` | optional conversion time |

Do a time-based split if timestamps exist:

\[
Train = \{i: t_i < T\}
\]

\[
Test = \{i: t_i \ge T\}
\]

This is better than a random split because production scoring predicts future leads from past data.

### Step 2: Train baseline models

Minimum models:

- random policy
- logistic regression
- random forest or balanced random forest
- gradient boosted trees, if available

The model output should be a score:

\[
\hat{s}_i \in [0,1]
\]

interpreted as:

\[
\hat{P}(Y_i=1\mid X_i)
\]

### Step 3: Rank test leads

Sort test leads by:

\[
\hat{s}_i
\]

Select:

\[
S_B^{score}
\]

for several values of \(B\), such as:

\[
B \in \{50,100,200,500\}
\]

### Step 4: Compare against random allocation

Random allocation selects \(B\) leads uniformly at random:

\[
S_B^{random}
\sim
Uniform(\{S: |S|=B\})
\]

The expected random conversion count is:

\[
E[\text{conversions@B}_{random}]
=
B \cdot \bar{Y}
\]

where:

\[
\bar{Y}=\frac{1}{n}\sum_i Y_i
\]

### Step 5: Evaluate top-K metrics

Precision@B:

\[
Precision@B
=
\frac{1}{B}\sum_{i\in S_B}Y_i
\]

Interpretation:

> Of the \(B\) selected leads, what fraction converted?

Recall@B:

\[
Recall@B
=
\frac{\sum_{i\in S_B}Y_i}{\sum_i Y_i}
\]

Interpretation:

> Of all converters in the test set, what fraction did the top \(B\) capture?

Capture Rate@B:

\[
Capture@B
=
\frac{\text{conversions in top } B}{\text{total conversions}}
\]

For this experiment, Capture@B is equivalent to Recall@B.

Lift@B:

\[
Lift@B
=
\frac{Precision@B}{\bar{Y}}
\]

Interpretation:

> How much denser are converters in the selected set compared with the average lead pool?

### Step 6: State the hypothesis

Null hypothesis:

\[
H_0:
Precision@B_{model}
\le
Precision@B_{random}
\]

Alternative hypothesis:

\[
H_1:
Precision@B_{model}
>
Precision@B_{random}
\]

Equivalent business version:

> The model-selected top \(B\) leads contain more converters than random selection.

### Step 7: Estimate uncertainty

Use bootstrap resampling on the test set.

For each bootstrap sample \(b\):

1. Resample test leads with replacement.
2. Recompute model Precision@B.
3. Recompute random Precision@B.
4. Compute:

\[
\Delta_b
=
Precision@B_{model,b}
-
Precision@B_{random,b}
\]

Then calculate:

\[
CI_{95\%}(\Delta)
\]

If the lower bound is above zero, the model ranking is statistically stronger than random selection at capacity \(B\).

## 7. What This Experiment Can and Cannot Prove

### It can prove

- whether a model ranks likely converters better than random
- whether the feature set contains useful predictive signal
- whether the system can create a useful sales queue

### It cannot prove

- that sales intervention caused conversion
- that selected leads have high uplift
- that contacting a lead increases revenue
- that the policy is optimal under intervention cost

The limitation is important:

\[
P(Y=1\mid X)
\ne
\mathbb{E}[Y(1)-Y(0)\mid X]
\]

A lead may have high conversion probability because they would convert organically. Later uplift experiments are needed to estimate incremental impact.

## 8. Previous Work and Useful References

### Lead scoring as ranking

Lead scoring is commonly defined as ranking prospects by their value or readiness for sales follow-up. This supports the practical framing of the experiment: the model is not only classifying leads, it is ordering them for action.

Reference:

- Wikipedia, "Lead scoring": https://en.wikipedia.org/wiki/Lead_scoring

Insight for this project:

> Lead scoring is naturally a ranking problem because sales teams act on a prioritized list, not on all leads equally.

### Precision and recall for ranking

Information retrieval evaluates ranked lists using precision and recall at cutoffs. This maps directly to lead allocation because sales capacity creates a cutoff \(B\).

References:

- Stanford IR Book, "Evaluation of ranked retrieval results": https://nlp.stanford.edu/IR-book/html/htmledition/evaluation-of-ranked-retrieval-results-1.html
- Evidently AI, "Precision and recall at K in ranking and recommendations": https://www.evidentlyai.com/ranking-metrics/precision-recall-at-k

Insight for this project:

> Precision@B and Recall@B are better aligned with sales operations than global accuracy because only the selected top \(B\) leads are contacted.

### Imbalanced classification

Rare-event prediction is difficult because standard learning algorithms and global metrics can be dominated by the majority class.

References:

- Haixiang et al., "Learning from class-imbalanced data: Review of methods and applications" (Expert Systems with Applications, 2017): https://www.sciencedirect.com/science/article/pii/S0957417416307175
- PLOS One, "An empirical evaluation of sampling methods for the classification of imbalanced data" (2022): https://journals.plos.org/plosone/article?id=10.1371/journal.pone.0271260
- Frontiers in Big Data, "Weighting Methods for Rare Event Identification From Imbalanced Datasets" (2021): https://www.frontiersin.org/articles/10.3389/fdata.2021.715320/full

Insight for this project:

> Accuracy is a weak metric for rare conversion tasks. Top-K precision, recall, lift, and PR-style metrics are more relevant.

### Scikit-learn metric tooling

Scikit-learn provides common classification and ranking-related metrics such as precision, recall, F-score, and average precision.

Reference:

- Scikit-learn metrics documentation: https://scikit-learn.org/stable/api/sklearn.metrics.html

Insight for this project:

> Standard metric tooling can support the baseline experiment, but custom top-K functions are needed because sales capacity is fixed by \(B\).

## 9. Final Experiment Statement

This experiment evaluates whether a standard conversion model can produce a better sales queue than random allocation.

The model estimates:

\[
\hat{P}(Y=1\mid X)
\]

and selects:

\[
S_B=\operatorname{TopB}_i(\hat{s}_i)
\]

The experiment is successful if:

\[
Precision@B_{model}
>
Precision@B_{random}
\]

with a positive confidence interval for the improvement.

This becomes the baseline that all later uplift and causal allocation experiments must outperform.
